legibility map

Anonymized, from a real engagement. This is what week two produces.

A legibility map answers one question for every part of a system: can a person still reason about this, or has it gone write-only — code that only ever gets appended to, because nobody can safely read what's already there.

Two measurements per module. How often it changes, pulled from the last quarter of git history. And how well anyone still understands it, scored from characterization tests and reading sessions with the people who own it.

write-only regions

Two modules a human can no longer safely read. Both change every week. Neither has a test that asserts what it's supposed to do. Stabilization starts here.

billing/InvoiceEngine

84 commits last quarter, most of them fixes to previous fixes. The method that decides what a customer owes is 340 lines with eleven nested branches on nullable fields. Two of your last three billing incidents originated in one untested branch.

orders/FulfillmentPipeline

A chain of handlers that pass a mutable bag of state between them. No handler documents what it expects to already be set, so every change is a guess about invisible order. The suite is green and asserts nothing about the actual sequence.
